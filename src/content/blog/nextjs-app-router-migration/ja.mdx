既存のNext.jsプロジェクトをPages RouterからApp Routerへ移行した。規模は中程度のSaaSアプリで、ページ数は30ほど。作業期間は2週間。やってみて初めてわかったことが多かった。

## 想定外だった部分

`getServerSideProps` と `getStaticProps` がなくなるのは知っていた。問題はその先だ。

サーバーコンポーネントとクライアントコンポーネントの境界をどこに引くかで、何度も設計をやり直した。`useState` や `useEffect` を使っているコンポーネントは自動的にクライアントコンポーネントになるが、それを親コンポーネントがサーバーコンポーネントとして扱えるか、どこに `'use client'` を書くかの判断が思ったより難しい。

`next-intl` のような i18n ライブラリも、App Router対応のバージョンへの移行が必要になった。サーバーコンポーネントでは `useTranslations` が使えず、`getTranslations` を使う必要があるなど、ライブラリ側の対応状況を一つひとつ確認する作業が発生した。

## Route Handler

Pages Routerの `pages/api` ディレクトリが `app/api` の Route Handler に変わる。

書き方は `Request` / `Response` オブジェクトを直接扱う形になり、Next.js独自のラッパーが薄くなった。型安全性は上がったが、`req.query` のパース処理など、これまで自動でやってくれていたものを自分で書く必要が出てくる場面がある。

## データフェッチの整理

`fetch` でキャッシュを制御できるようになったのは良かった。

`{ cache: 'force-cache' }` で静的、`{ cache: 'no-store' }` で動的、`{ next: { revalidate: 60 } }` でISR相当の動作をコンポーネントレベルで設定できる。ページ単位でしか制御できなかった以前より粒度が細かくなった。

## やって良かった判断

移行を段階的にやったのは正解だった。

App RouterとPages Routerは同じプロジェクト内で共存できる。まず影響の少ない静的ページから移行し、動的ページ・認証まわり・APIルートの順に進めた。一気にやらなかったことで、各段階での問題を切り分けられた。

## 結論

App Routerへの移行は、単なる書き換え作業ではなく、アーキテクチャの再設計に近い。サーバーとクライアントの責務を明示的に分ける思想を理解していないと、動くけど設計が崩れたコードになる。

慣れると生産性は上がる。ただし、慣れるまでのコストは相応にある。
