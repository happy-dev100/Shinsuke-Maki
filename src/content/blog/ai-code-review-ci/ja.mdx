コードレビューの一部をAIに任せる実験を、実際のプロジェクトで試してきた。結論から言うと「使える部分」と「使えない部分」がはっきり分かれる。

## 実装の概要

GitHub ActionsのPRトリガーで起動し、差分を取得してOpenAI APIに渡す。返ってきたレビューコメントをPRのコメントとして投稿する。これだけだ。

```yaml
on:
  pull_request:
    types: [opened, synchronize]
```

差分はgitの`--unified=0`オプションで変更部分のみ取得し、トークン数を節約する。ファイル単位で処理して、大きすぎるファイルはスキップする。

## 使えた部分

明らかなミスの検出は精度が高い。

- `await` の付け忘れ
- nullチェックの漏れ
- 不要な`console.log`の残留
- 変数名の一貫性のなさ
- 型アノテーションの欠落

これらは人間がレビューするより速く、かつ見落とさない。単純なルールベースのlintで取れないものを拾ってくれる場面もある。

## 使えなかった部分

設計レベルの指摘は信頼できない。

「このアーキテクチャは将来的にスケールしない」「このデータ構造は要件に合っていない」といった判断は、コードの差分だけを見ていてはできない。コンテキストが不足している状態で、もっともらしい指摘を作り出すことがある。

ビジネスロジックの正しさの検証も同様だ。

## ノイズとの付き合い方

AIのレビューは必ずノイズを含む。関係ないコメント、的外れな指摘、重複した内容——これらが混じる。

ノイズを許容できるかどうかは、チームとプロジェクトによる。「参考程度に見る」という運用であれば問題ない。「全コメントに対応する」という運用にすると、かえって開発速度が落ちる。

## 現在の使い方

レビュアーが人間の場合の「事前チェック」として機能させている。

人間がレビューする前にAIが明らかなミスを潰しておくことで、レビュアーが設計・ロジック・品質に集中できる。AIを「レビュアーの代替」ではなく「レビュアーの前処理」として使う位置づけが現時点での正解だと思っている。
